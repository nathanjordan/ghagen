# 0005 — Typed engine-report seam + dogfooded check-deps

Status: Part A implemented (typed report contract + `helper_provided` removal); Part B implemented (dogfooded `check-deps` via `ghagen_workflows.py`)

## 1. Problem

### 1.1 The phantom `helper_provided` field

`check-deps/action.yml` (hand-written composite action, ~260 lines of Bash +
inline Python heredocs) consumes `ghagen deps upgrade --json`. It reads a
`helper_provided` key at four sites:

```bash
# action.yml:75, 80  (Detect step)
HELPER_PROVIDED=$(python3 -c "import json; d=json.load(open('$JSON_FILE')); print(len(d.get('helper_provided',[])))")
echo "helper_provided=$HELPER_PROVIDED" >> "$GITHUB_OUTPUT"
```

```python
# action.yml:179-185 (PR body) and 251-257 (issue body)
helpers = d.get('helper_provided', [])
if helpers:
    lines.append('### Helper-provided actions (requires ghagen upgrade)')
    for h in helpers:
        lines.append(f'- `{h["uses"]}` -> `{h["latest"]}` (upgrade ghagen to get this)')
```

**No engine in either port emits `helper_provided`.** The upgrade JSON payload
is produced from `UpgradeReport`, whose only report lists are `version_bumps`
and `lockfile_stale`:

- Python: `pin/engine.py` `UpgradeReport` → serialized by `cli/deps.py`
  `_bump_to_json` / `_stale_to_json` (lines 223-245); the assembled dict has
  only `version_bumps` / `lockfile_stale` keys (lines 208-218).
- TypeScript: `pin/engine.ts` `UpgradeReport { versionBumps, lockfileStale,
changedFiles, warnings }` → `cli/deps.ts` `bumpToJson` / `staleToJson`
  (lines 209-233); same two output keys.

The consumer only "works" because every read is `.get('helper_provided', [])`,
which silently returns `[]`. The `helper_provided` output, the `### Helper-provided`
PR/issue sections, and the "upgrade ghagen to get this" copy are dead — they can
never fire. This is a seam that **already silently drifted**: the emitter dropped
(or never grew) the key, and nothing failed.

### 1.2 The seam is untyped, unversioned, untested

`ghagen deps upgrade --json` is a contract between the CLI (producer) and
`check-deps/action.yml` (consumer), but:

- Its shape is asserted nowhere. `grep` for `--json` / `version_bumps` in the
  test suites finds **no** test that pins the rendered JSON output in either
  port. `test_cli/test_deps.py` (97 lines) never exercises `--json`; TypeScript
  has no `cli/deps.test.ts` at all.
- It has no schema and no version. The producer's field names live only in the
  serializer functions; the consumer's expected field names live only in Bash
  string literals. Nothing links them.
- Drift is invisible. A rename like `latest → latest_tag` would break the action
  at runtime with zero signal from CI, exactly as `helper_provided` already did.

### 1.3 Hand-maintained vs. dogfooded asymmetry

ghagen ships two composite actions. They are maintained in opposite ways:

|                     | `check-synth/action.yml`                                                                                                              | `check-deps/action.yml`                                                                           |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Source of truth     | `.github/ghagen_workflows.py` `_ghagen_check_action()` (line 465), registered via `app.add_action(..., dir="check-synth")` (line 537) | the YAML file itself, edited by hand                                                              |
| Generated by ghagen | yes (header: "generated by ghagen … Do not edit manually")                                                                            | no                                                                                                |
| Refs pinned to SHAs | yes — `actions/setup-python@ece7cb06…  # v6` via the pin transform                                                                    | no — `actions/setup-python@v6` (floating tag)                                                     |
| Verified in CI      | yes — `ghagen check-synced` (ci.yml check-sync job) diffs it, and the `test-action` job runs `uses: ./check-synth`                    | **no** — `check-deps` is referenced by nothing: not by `ghagen_workflows.py`, not by any workflow |

`check-deps` is an orphan: unpinned, ungenerated, unverified. The pin transform
(`app._build_transforms`) already resolves `actions/setup-python@v6` — the
`.ghagen.lock.yml` even holds its SHA (`ece7cb06…`) — but `check-deps` never
passes through synthesis, so it never receives the pin.

## 2. Design — Part A: typed report contract

### 2.1 The engine reports are the source of truth

`UpgradeReport` (both ports) already is a typed structure. The fix is to make
the **serialized JSON shape** a checked artifact derived from it, and to delete
the consumer's phantom handling. No new field is invented; `helper_provided`
simply never existed and is removed from the reader.

### 2.2 Canonical JSON schema

Derived verbatim from `VersionBump` / `LockfileStaleEntry` and the serializers:

```json
{
  "version_bumps": [
    {
      "uses": "actions/checkout@v5",
      "current": "v5",
      "latest": "v6",
      "severity": "major",
      "source_files": [".github/ghagen_workflows.py"]
    }
  ],
  "lockfile_stale": [
    {
      "uses": "actions/setup-python@v6",
      "current_sha": "ece7cb06caefa5fff74198d8649806c4678c61a1",
      "latest_sha": "aaaa1111bbbb2222cccc3333dddd4444eeee5555",
      "source_files": [".github/ghagen_workflows.py"]
    }
  ]
}
```

Invariants (all already true in code; the tests below freeze them):

- `severity ∈ {"major","minor","patch"}` (the `Severity`/`BumpSeverity` literal).
- `source_files` is **omitted** when empty; a list of strings otherwise.
- `version_bumps` is present only when `mode ∈ {versions, all}`; `lockfile_stale`
  only when `mode ∈ {lockfile, all}` (deps.py:210-217 / deps.ts:196-201).
- **No `helper_provided`, ever.**
- Edge case to lock down: the "everything up to date" early return
  (deps.py:196-206 / deps.ts:180-189) emits _both_ keys as `[]` regardless of
  `mode`. Tests must assert this exactly (or the code is changed to honor `mode`
  in the empty case too — pick one and pin it).

### 2.3 Where it is asserted

Follow the existing cross-port snapshot infrastructure. Shared golden fixtures
already live at the repo-root `fixtures/expected/*.yml` and are loaded by **both**
ports: Python `tests/test_integration/` (`conftest.py` → `FIXTURES_DIR`) and
TypeScript `src/integration/test-utils.ts` (`loadFixture`, same
`../../../../fixtures` path). Snapshot tests build a model in each language and
`assert rendered == loadFixture(name)` — parity by construction against one file.

Reuse that pattern for the JSON seam:

1. **Shared golden fixture.** Add `fixtures/expected/upgrade_report.json`
   containing a representative payload (one version bump with `source_files`,
   one stale entry, exercising the omit-when-empty rule via a second bump with
   no `source_files`).
2. **Per-port serializer test.** In each port, construct an `UpgradeReport` with
   the same fixed data and assert the rendered `--json` string parses to the
   fixture JSON:
   - Python `tests/test_cli/test_deps.py`: build the report, call the
     serializer path, `assert json.loads(out) == json.loads(fixture_text)`.
   - TypeScript new `src/cli/deps.test.ts`: same against `loadFixture(
"upgrade_report.json")`.
     Because both ports assert against the **same** file, they are parity-checked
     transitively — the exact mechanism the YAML snapshots already use, so no new
     test infrastructure is required.
3. **Negative assertion.** Both tests assert `"helper_provided" not in output`,
   so re-introducing the phantom key fails CI.

Optional hardening (nice-to-have, not required): a JSON-Schema snapshot under
`schema/` (sibling to `action_schema.json`) validated in each port via the
existing jsonschema/ajv harness. Deferred unless cheap.

### 2.4 Removal of `helper_provided` from the action

Delete all four sites: the `HELPER_PROVIDED` computation and its `$GITHUB_OUTPUT`
line (action.yml:75, 80), and the two `### Helper-provided` heredoc blocks
(179-185, 251-257). This change lands as part of the Part-B rewrite (the step
bodies move into Python), so the phantom code is deleted rather than migrated.

> **Deviation (Part A implementation note):** this spec assumed the removal
> would land only as part of the Part-B rewrite. Part A removes the four
> `helper_provided` sites directly from the hand-written
> `check-deps/action.yml` instead (no behavior change beyond deleting dead
> code), so the JSON contract is enforced on the actual consumer immediately
> rather than waiting on Part B. While editing those same heredoc blocks, a
> **pre-existing, unrelated bug** was also found and fixed: the two
> `python3 -c "..."` blocks (PR-body and issue-body builders) embedded raw,
> zero-indented Python source inside a YAML block literal scalar (`run: |`)
> whose established indentation was 6 spaces. Per the YAML spec, a
> less-indented non-empty line terminates a block literal scalar early, which
> made `check-deps/action.yml` invalid YAML (confirmed with PyYAML, ruamel.yaml,
> and the JS `yaml` package — all three fail identically at that line, both
> before and after this change). Because `check-deps` is an orphan (§1.3,
> nothing in CI ever parses it), this had never been caught. Fixed by
> uniformly indenting every non-blank line of both embedded Python blocks by 6
> spaces (preserving the scripts' relative indentation, hence their semantics)
> so they satisfy the block scalar's indentation requirement; verified by
> parsing with all three YAML libraries, `py_compile`-checking the extracted
> Python (after simulating bash's `\"`/`` \` `` unescaping), and an end-to-end
> run of the PR-body heredoc. Part B's full rewrite into
> `ghagen_workflows.py`/triple-quoted `run` strings removes this class of bug
> entirely.

## 3. Design — Part B: dogfood `check-deps`

Define `check-deps` in `.github/ghagen_workflows.py` exactly as `check-synth` is
defined, so it enters the generate-and-check loop and receives pinned refs.

### 3.1 Before (hand-written YAML excerpt)

```yaml
# check-deps/action.yml (current, hand-maintained)
runs:
  using: composite
  steps:
    - uses: actions/setup-python@v6 # floating, unpinned
      with:
        python-version: ${{ inputs.python-version }}
    - name: Install ghagen
      run: |
        if [ -n "${{ inputs.ghagen-version }}" ]; then
          pip install "ghagen==${{ inputs.ghagen-version }}"
        else
          pip install ghagen
        fi
      shell: bash
    - name: Detect outdated dependencies
      id: detect
      env:
        GITHUB_TOKEN: ${{ inputs.token }}
      run: |
        set -euo pipefail
        JSON_FILE=$(mktemp)
        ghagen deps upgrade --check --json --mode "${{ inputs.mode }}" --config "${{ inputs.config }}" > "$JSON_FILE" || true
        # ... version_bumps / lockfile_stale parsing (helper_provided lines removed) ...
      shell: bash
  # ... Create PR / Create issue steps ...
```

### 3.2 After (proposed Python definition)

Mirror `_ghagen_check_action()` (line 465). All inputs (`mode`, `output`,
`config`, `python-version`, `ghagen-version`, `token`, `labels`, `branch-prefix`,
`commit-message-prefix`, `group`) become `ActionInput`s; each Bash/Python step
body becomes a triple-quoted `run` string — `auto_dedent` (App default) strips
the indentation and the `<<'PY'` heredocs survive unchanged, as they already do
for the `homebrew-bump` job in `_release_workflow()`.

```python
def _ghagen_update_action() -> Action:
    """ghagen's own composite action wrapping `ghagen deps upgrade`.

    Dogfooding: replaces the hand-written check-deps/action.yml so it is
    generated, pinned, and drift-checked like check-synth.
    """
    return Action(
        name="ghagen Update",
        description=(
            "Detect and apply dependency updates for "
            "ghagen-managed GitHub Actions workflows"
        ),
        branding=Branding(icon="refresh-cw", color="blue"),
        inputs={
            "mode": ActionInput(
                description="Detection mode: 'versions', 'lockfile', or 'all'",
                required=False, default="all",
            ),
            "output": ActionInput(
                description="Output type: 'pr' or 'issue'",
                required=False, default="pr",
            ),
            "config": ActionInput(
                description="Path to ghagen config file",
                required=False, default=".github/ghagen_workflows.py",
            ),
            "python-version": ActionInput(
                description="Python version to use",
                required=False, default="3.13",
            ),
            "ghagen-version": ActionInput(
                description="ghagen version to install (empty for latest)",
                required=False, default="",
            ),
            "token": ActionInput(
                description="GitHub token for API calls and PR/issue creation",
                required=False, default="${{ github.token }}",
            ),
            "labels": ActionInput(
                description="Comma-separated labels to apply to PRs/issues",
                required=False, default="",
            ),
            "branch-prefix": ActionInput(
                description="Branch name prefix for PRs",
                required=False, default="ghagen-update/",
            ),
            "commit-message-prefix": ActionInput(
                description='Optional prefix for commit messages (e.g. "chore(deps):")',
                required=False, default="",
            ),
            "group": ActionInput(
                description="Group all updates into a single PR/issue",
                required=False, default="false",
            ),
        },
        runs=CompositeRuns(
            steps=[
                Step(
                    uses="actions/setup-python@v6",  # pin transform -> SHA at synth
                    with_={"python-version": "${{ inputs.python-version }}"},
                ),
                Step(
                    name="Install ghagen",
                    run="""
                        if [ -n "${{ inputs.ghagen-version }}" ]; then
                          pip install "ghagen==${{ inputs.ghagen-version }}"
                        else
                          pip install ghagen
                        fi
                    """,
                    shell="bash",
                ),
                Step(
                    name="Detect outdated dependencies",
                    id="detect",
                    env={"GITHUB_TOKEN": "${{ inputs.token }}"},
                    run="""
                        set -euo pipefail
                        JSON_FILE=$(mktemp)
                        ghagen deps upgrade --check --json --mode "${{ inputs.mode }}" --config "${{ inputs.config }}" > "$JSON_FILE" || true
                        VERSION_BUMPS=$(python3 -c "import json; d=json.load(open('$JSON_FILE')); print(len(d.get('version_bumps',[])))")
                        LOCKFILE_STALE=$(python3 -c "import json; d=json.load(open('$JSON_FILE')); print(len(d.get('lockfile_stale',[])))")
                        echo "json_file=$JSON_FILE" >> "$GITHUB_OUTPUT"
                        echo "version_bumps=$VERSION_BUMPS" >> "$GITHUB_OUTPUT"
                        echo "lockfile_stale=$LOCKFILE_STALE" >> "$GITHUB_OUTPUT"
                        TOTAL=$((VERSION_BUMPS + LOCKFILE_STALE))
                        echo "total_updates=$TOTAL" >> "$GITHUB_OUTPUT"
                        # ... (helper_provided lines deleted) ...
                    """,
                    shell="bash",
                ),
                # Step "Exit early if no updates"  (if_=...)
                # Step "Create PR"    (if_=..., PR-body heredoc, helper block removed)
                # Step "Create issue" (if_=..., issue-body heredoc, helper block removed)
            ],
        ),
    )
```

Register it alongside the existing action:

```python
def create_app() -> App:
    app = App()
    ...
    app.add_action(_ghagen_check_action(), dir="check-synth")
    app.add_action(_ghagen_update_action(), dir="check-deps")   # new
    return app
```

Fidelity notes:

- `Step` supports `id`, `if_`, `env`, `with_`, `shell` — every field the current
  YAML uses. The `Create PR` / `Create issue` steps carry their `if:` guards
  (e.g. `steps.detect.outputs.total_updates != '0' && inputs.output == 'pr'`)
  verbatim as `if_=`.
- The action exposes no action-level `outputs:`; it only uses per-step
  `$GITHUB_OUTPUT` + `steps.detect.outputs.*`, so no `ActionOutput` is needed.
- The Python-heredoc PR/issue-body builders port unchanged **minus** the
  `helper_provided` blocks (§2.4). `auto_dedent` + the `<<'PY'` sentinel keep
  the heredoc bodies intact (same mechanism as `_release_workflow`'s
  `python3 - <<'PY'` block, already round-tripping in CI).

### 3.3 How CI catches drift (same as check-synth)

Once registered, `check-deps/action.yml` becomes a synthesized item:

- `ghagen check-synced` (`app.check()`, app.py:173) diffs the on-disk YAML
  against the freshly rendered model for **every** registered item, actions
  included. The ci.yml `check-sync` job (`uv run ghagen check-synced`) and the
  lint job's `ghagen deps check-synced` now fail if `check-deps/action.yml` is
  edited by hand or left stale — identical to check-synth today.
- The pin transform runs during synth for `item_type == "action"`
  (`app._apply_transforms`), so `actions/setup-python@v6` is rewritten to its
  locked SHA (`ece7cb06…`, already in `.ghagen.lock.yml`). check-deps loses its
  unpinned ref for free.
- The generated file gains the "Do not edit manually" header, closing the
  hand-edit loophole.

## 4. Migration — order of operations

1. **Fix the phantom field first (Part A, independent, small).**
   a. Add `fixtures/expected/upgrade_report.json`.
   b. Add the serializer tests (Python `test_deps.py`, new TS `deps.test.ts`),
   including the `helper_provided` negative assertion. These pass immediately
   against current serializers (no `helper_provided` is emitted), locking the
   contract before any consumer change.
2. **Dogfood conversion (Part B).**
   a. Author `_ghagen_update_action()` + register in `create_app()`.
   b. Run `ghagen synth` (or let the next generation run) to regenerate
   `check-deps/action.yml`; verify the diff is only the ghagen header + the
   setup-python SHA pin + the removed `helper_provided` lines + formatting.
   c. `ghagen check-synced` must go green.

Part A is safe to land alone and de-risks Part B (the JSON shape is frozen before
the consumer is rewritten).

## 5. Test plan

- **Python** `tests/test_cli/test_deps.py`: build an `UpgradeReport` fixture,
  render `--json`, assert `json.loads(out) == fixture` and `"helper_provided"
not in out`; assert `source_files` omitted-when-empty; assert `--mode versions`
  omits `lockfile_stale` and vice-versa; assert the empty-report early-return
  shape.
- **TypeScript** new `src/cli/deps.test.ts`: same assertions against
  `loadFixture("upgrade_report.json")`.
- **Cross-port parity:** both ports assert against the single shared
  `fixtures/expected/upgrade_report.json`, mirroring the YAML snapshot harness.
- **Action generation:** the existing `snapshot`/integration action tests plus
  `ghagen check-synced` cover the regenerated `check-deps/action.yml`; add it to
  whatever fixture/snapshot enumeration lists composite actions if one is
  exhaustive. The `test-action` CI job may optionally gain a smoke invocation of
  `uses: ./check-deps` (dry-run `--check`), matching `./check-synth`.
- **Regression guard:** deleting the negative `helper_provided` assertion or
  re-adding the key fails CI.
